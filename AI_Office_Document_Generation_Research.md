# AI 办公文档生成方案调研：MCP / Skill 扩展的不足

> 2026-06-05 | 聚焦 AI 幻觉 + 背景知识缺失 + MCP/Skill 架构局限性

---

## 一、当前格局

2025-2026 年，办公 AI 分为两类：

| 类型 | 代表产品 | 模式 | 文档生成方式 |
|------|----------|------|-------------|
| **通用桌面助手** | QoderWork (阿里)、WorkBuddy (腾讯)、DuMate (字节) | 桌面 Agent → 操作本地文件 | Skill + python-pptx/python-docx 脚本生成 |
| **专项生成工具** | AiPPT、百度文库 PPT、Gamma | 云端生成 → 模板驱动 | 专有模型 + 结构化模板 |

**本文聚焦第一类**——它们通过 MCP 协议或 Skill 系统扩展 LLM 能力，试图覆盖文档+PPT 生成。问题在于：这些扩展机制本质上解决的是"能做什么"（能力接入），而非"做的对不对"（质量保障）。

---

## 二、MCP / Skill 扩展的架构局限

### 2.1 RAG vs Skill vs MCP 的本质差异

Alex Ewerlöf (2026) 的精辟分类：

| 技术 | 核心问题 | 解决方式 | 文档生成的对应 |
|------|----------|----------|---------------|
| **RAG** | "LLM 知道什么" | 检索 → 注入上下文 | 喂给 LLM 公司资料、历史文档作为参考 |
| **Skill** | "LLM 能做什么" | 按需加载能力脚本 | python-pptx 生成命令、模板选择逻辑 |
| **MCP** | "LLM 如何安全调用" | 标准化工具接口 | 连接文件系统、数据库、SaaS API |

**关键矛盾**：MCP 和 Skill 解决的是**工具接入层**的问题——让 LLM 能调用 python-pptx、读写文件、连接数据库。但它们**完全不解决内容质量层**的问题——生成的内容是否准确、是否基于真实数据、是否符合品牌规范。

### 2.2 Skill 模式的"盲生成"问题

以 WorkBuddy 生成 PPT 的典型流程为例：

```
用户输入: "帮我生成一份Q2销售周报PPT"

Skill 执行链:
  1. LLM 理解意图 → 拆解为步骤
  2. Skill Router → 加载 PPT Skill
  3. LLM 生成内容（文本、数据、结构）  ← 纯凭模型记忆
  4. python-pptx 脚本执行 → 写入 .pptx 文件
  5. 返回文件给用户
```

**问题出在第 3 步**：LLM 生成 PPT 中的「销售数据」「增长趋势」「竞品对比」等核心内容时，依赖的是训练数据中残留的模糊记忆，而非你公司的真实数据。结果可能是：

- ❌ 虚构了不存在的 Q2 销售数字（AI 幻觉）
- ❌ 引用了行业平均数据而非你的实际数据
- ❌ 使用了过时的产品名称/品牌名称
- ❌ 错判了公司部门结构（"销售一部" vs 实际叫"华东大区"）

### 2.3 MCP 的上下文膨胀问题

MCP 协议定义的"工具"是一个完整的 JSON schema + 描述。当 LLM 接入 10+ 个 MCP 工具时：
- 每个工具的描述占用 500-2000 token
- LLM 的上下文窗口被工具描述填满
- 留给「实际业务内容」的窗口大幅缩水
- 生成长文档时 LLM 容易"忘了前面说了什么"

**实测案例**：WorkBuddy 处理 10+ 页文档时经常"只输出原文档的一部分"，长文档处理卡壳——本质上是上下文窗口被 MCP 工具描述挤占，留给内容推理的空间不足。

## 三、AI 幻觉在文档生成中的 7 类典型表现

根据 WorkBuddy、QoderWork 的用户实际反馈和学术研究，办公文档生成中的幻觉分为以下 7 类：

### 第 1 类：数值幻觉（最危险）

| 表现 | 示例 | 后果 |
|------|------|------|
| 虚构统计数据 | PPT 中写"Q2 营收 3800 万，同比增长 23%" | 老板拿去汇报，被当场拆穿 |
| 捏造财务数字 | 报告中列"毛利率 41.2%，净利润 850 万" | 法务/合规风险 |
| 凭空生成表格数据 | Excel 报表中填充了不存在的销售明细 | 决策依据错误 |

### 第 2 类：引用幻觉

| 表现 | 示例 |
|------|------|
| 虚构文献引用 | 报告中引用"据 McKinsey 2025 报告显示……"——McKinsey 从未发表该报告 |
| 捏造权威背书 | "根据 Gartner Magic Quadrant……"——实际排名与此不同 |
| 虚假数据来源 | "国家统计局数据显示……"——数字是 LLM 编的 |

### 第 3 类：实体幻觉

| 表现 | 示例 |
|------|------ |
| 虚构产品名 | PPT 中出现"OpenCopilot Pro Max 企业版"——这个产品不存在 |
| 错用品牌名 | 报告把"飞书"写成"钉钉"，把"通义千问"写成"文心一言" |
| 捏造人名/职衔 | "CEO 张伟表示……"——你们公司根本没有叫张伟的 CEO |

### 第 4 类：逻辑幻觉

| 表现 | 示例 |
|------|------|
| 因果关系颠倒 | "因利润率提升，导致成本下降"——逻辑反了 |
| 时间线混乱 | 把 Q1 的事件放在 Q3 时间线上 |
| 矛盾陈述 | 前一页说"团队 15 人"，后一页说"团队 35 人"——没有解释 |

### 第 5 类：格式幻觉

| 表现 | 示例 |
|------|------|
| 布局错乱 | 文本溢出文本框、图片重叠、表格断页 |
| 字体丢失 | 生成时指定了中文字体，但在无该字体的环境下打开全部变方块 |
| 重复/空白页 | 生成 12 页 PPT，其中 1 页空白、2 页内容几乎相同 |

**WorkBuddy 用户真实反馈**：
> "我找 WorkBuddy 帮忙说'帮我生成这份 PPT'，它很积极噼里啪啦调了 python-pptx，生成文件我打开一看——布局错乱、内容挤成一团、字体全部丢失，反复修改了四次，每次都是废的。"

### 第 6 类：风格幻觉

| 表现 | 示例 |
|------|------|
| 不符合品牌调性 | 公司走严肃 B2B 路线，PPT 却用卡通 emoji 和俏皮话 |
| 不符合受众预期 | 给 CXO 的汇报用了太多技术 jargon |
| 语体不一致 | 一页正式书面语，下一页突然口语化 |

### 第 7 类：知识截止幻觉

| 表现 | 示例 |
|------|------|
| 过时信息 | LLM 训练截止 2024 年 6 月，但 PPT 讨论 2025 Q4 业务 |
| 已变更的政策 | 报告中引用了已被废除的法规 |
| 不存在的功能 | 分析产品时提到了还没发布的功能 |

## 四、背景知识缺失导致的深层问题

### 4.1 无法获取"只能从内部知道"的信息

办公文档（特别是 PPT）的核心价值在于「基于公司真实数据的洞察」。MCP/Skill 可以让 AI 读文件、搜网页，但以下信息是无论怎么打 MCP 工具都拿不到的：

| 缺失的信息类型 | 为什么拿不到 | 导致的问题 |
|---------------|-------------|-----------|
| **公司内部数据** | CRM 数据、财务系统数据、ERP 数据没有 MCP 接口 | 只能编造或使用行业平均数据 |
| **团队组织架构** | 无标准 API 获取"华东大区到底有几个人" | 人员分配/任务分解错得离谱 |
| **品牌规范** | Logo、配色、字体、VI 手册没有结构化存储 | 生成的 PPT 风格像别家公司的 |
| **上下文历史** | 上次汇报的结论、上次会议的共识 | 前后矛盾、推翻已有决策 |
| **行业黑话** | 公司内部缩写、产品代号、业务术语 | 用了错误的或不专业的表达 |
| **战略方向** | OKR、年度目标、重点项目优先级 | 建议的方案和公司方向背离 |

### 4.2 RAG 方案的"表面接地"

理论上，可以通过 RAG 把公司文档灌给 LLM 解决背景知识问题。但实践中：

1. **分块策略粗糙**：把 300 页的产品手册切块投喂，LLM 可能检索到第 2 页和第 287 页的片段，但两页之间 285 页的上下文完全丢失
2. **检索失败静默**：RAG 检索可能返回了不相关的内容（语义相似但业务无关），LLM 基于这些错误上下文"自信地"生成了错误内容
3. **时效性问题**：知识库是静态快照，不会实时更新（即使加了实时检索，也有同步延迟）
4. **非结构化数据的困境**：公司资料大多是 PDF、邮件、IM 聊天记录，缺乏结构化标注

**RAG + LLM 拼图模式的核心问题**：
> "检索到的内容是碎片，LLM 用自己的幻觉把碎片粘起来，看起来像个完整的拼图——直到你仔细看才发现边缘对不上。"

## 五、当前产品的实测痛点汇总

### 5.1 WorkBuddy（腾讯）

| 痛点 | 严重度 | 根因 |
|------|:---:|------|
| PPT 生成质量一般，模型版本不是最新 | 🔴 | Skill 调用的 LLM 版本未统一 |
| 长文档处理卡壳，只输出部分内容 | 🔴 | 上下文窗口被 MCP 工具描述 + 长文档内容挤满 |
| 插件生态不成熟，安装常报错 | 🟡 | Skill 分发/安装机制不完善 |
| 生成 PPT 布局错乱、字体丢失 | 🔴 | python-pptx 对复杂格式支持弱 |
| 模糊指令理解偏差 | 🟡 | LLM 推理在弱约束下容易发散 |
| 复杂多步骤任务中途卡死 | 🔴 | Skill 链式执行缺少 checkpoint 和故障恢复 |

### 5.2 QoderWork（阿里）

| 痛点 | 严重度 | 根因 |
|------|:---:|------|
| 输出的 PPT 实质上是 HTML 网页，不是真正的 pptx | 🟡 | 产品定位偏向生成"能看的东西" |
| 生成内容"更像摘要而非简报"，信息量薄 | 🔴 | LLM 对具体业务细节缺乏知识 |
| 存在空白页（生成 7 页有 1 页空白） | 🟡 | Skill 执行缺少质量检查 |
| 需要二次加工（复制到 PPT 手动排版） | 🟡 | 输出格式和用户预期不匹配 |
| 执行速度在不同任务间波动大（6-10 分钟） | 🟡 | 多 Skill 协调缺少优化 |

### 5.3 DuMate（字节）

| 痛点 | 严重度 | 根因 |
|------|:---:|------|
| PPT 内容中规中矩，缺乏深度分析 | 🟡 | LLM 缺乏特定业务领域知识 |
| 无品牌定制能力 | 🔴 | 无法接入企业内部样式/模板系统 |
| Office 文件操作虽快，但复杂排版不如手工 | 🟡 | 自动化 vs 精细度的天然矛盾 |

### 5.4 共性缺陷

| 问题 | 影响 |
|------|------|
| **无验证闭环** | 生成的 PPT/文档没有自动事实核查环节 |
| **无版本对比** | 改了几轮后不知道哪版是正确的 |
| **无人介入点** | AI 自主生成了全文，用户难以在中间插入修改意图 |
| **无领域锚点** | 生成前不清楚"这是给哪个行业/哪种场景" |
| **代码执行脆弱** | python-pptx/python-docx 对中文、复杂排版、跨平台字体支持不足 |

## 六、OpenCopilot 的对标策略

基于以上分析，OpenCopilot 在设计 PPT/文档生成能力时，应走一条不同的路径——不是"AI 替你生成"，而是"AI 在你旁边帮你生成"。

### 6.1 差异化定位

| | QoderWork / WorkBuddy | OpenCopilot 方案 |
|---|---|---|
| 交互模型 | "帮我生成一份 PPT" → AI 自主输出 | "我选中这段内容 → AI 给我写好这一页" |
| 人在回路 | 事后审核（整份做完再看） | 实时参与（每页/每段确认） |
| 数据来源 | LLM 记忆 + 可能 RAG | 用户正在操作的文档内容 + 系统级上下文 |
| 幻觉防护 | 无主动防护 | Pipeline ImmuneSystem 层拦截危险输出 |
| 格式生成 | python-pptx 脚本 | 用户已在 Office/编辑器中有模板，AI 只生成内容 |

### 6.2 核心设计原则

1. **上下文就近原则**：不要问用户"你的数据在哪"，而是直接读取用户当前正在操作的软件（Word/PPT/Excel）中的真实内容
2. **增量而非全量**：不是生成整份 PPT，而是在用户创建了结构后，逐页辅助填充内容
3. **可验证的输出**：对数据类断言标注置信度（✅ 确认 / ⚠️ 需验证 / ❓ 推测），用户可以立即判断是否需要核实
4. **模板优先而非生成优先**：用户的 PPT 模板已经在 Office 里打开了——AI 只是帮填内容，不做格式

### 6.3 需要补充的能力

| 能力 | 当前状态 | 说明 |
|------|:---:|------|
| 品牌知识库 | ❌ 未实现 | 加载公司 Logo、配色、字体、VI 规范作为生成约束 |
| 内部数据源接入 | ❌ 未实现 | CRM/ERP/数据库的 MCP 连接器 |
| 输出置信度标注 | ❌ 未实现 | AI 生成内容的可信度分级+来源引用 |
| 模板系统 | ❌ 未实现 | 用户自定义 PPT/文档模板，AI 按模板填充 |
| 多轮修订闭环 | ✅ 已实现 | Pipeline 的 Agent Loop 支持 Plan-and-Solve/ReAct |
| 上下文感知 | ✅ 已实现 | Broker AXUIElement + IDE Extension 多源读取 |
| Pipeline 安全拦截 | ✅ 已实现 | ImmuneSystem + SecurityGuard 多层防护 |

## 七、结论

当前的办公 AI 通过 MCP/Skill 扩展已经解决了"能调用 python-pptx、能生成 .pptx 文件"的**工具层**问题。但在**内容质量层**，它们面临两个根本性瓶颈：

1. **AI 幻觉无法根除**：LLM 的本质是概率生成，不是事实数据库。数值幻觉、引用幻觉、实体幻觉在无人工核对的场景下必然出现。RAG 方案可以缓解（把真实知识灌进去），但检索失败静默 + 分块粗糙导致"看起来接地气、实际上张冠李戴"。

2. **背景知识天然缺失**：公司内部数据（CRM、财务、OKR）、品牌规范、团队架构、行业黑话——这些信息不在任何公开的 MCP 工具里，也不是 RAG 能检索到的结构化数据。

**朴素结论**：2026 年的办公 AI 在文档生成上，更适合做"辅助打稿 + 人工复核"的半自动模式，而非"AI 自主生成 + 一键交付"的全自动模式。后者在不需要准确性的场景（如 brainstorming 草稿）可以接受，但在有商务严肃性的场景（如对外汇报 PPT、合同、招标文件）存在不可接受的风险。

OpenCopilot 的「系统级按需介入 + 人在回路」的交互模式，恰好是一种更务实的路径——AI 出现在用户操作软件的时刻，帮忙填一页、改一段、查一条数据，但最终每个决策由人确认。
