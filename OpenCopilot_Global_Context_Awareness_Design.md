# OpenCopilot 全局应用上下文感知设计与扩展路径

> **日期**：2026-05-30  
> **主题**：守护进程监听全局应用切换行为的潜在价值与扩展路线图

本文档旨在探讨目前守护进程（Broker/Daemon）监听用户全局行为（如应用启停、应用间切换）的能力，并规划如何利用该能力将 AI 从“被动响应”升级为“主动感知上下文”的个人操作系统级助理（OS-Level Copilot）。

---

## 一、 核心场景：它能解决什么痛点？

频繁的应用切换往往意味着“工作流的割裂”或“意图的转移”。基于此，我们可以构建以下高价值场景：

### 1. 跨应用上下文感知与无缝衔接 (Context-Awareness)
- **场景**：用户在浏览器中复制了一段报错信息，随后切换到了 IDE。
- **AI行为**：守护进程捕获到 `Browser -> IDE` 的切换动作，AI 助理可以主动在 IDE 的侧边栏或悬浮窗提示：“检测到您刚在浏览器查阅了报错，需要我为您在当前代码中定位并修复这个问题吗？”
- **价值**：消除用户手动描述背景信息的步骤，实现跨端数据的隐式打通。

### 2. 动态工作流意图预测 (Intent Prediction)
- **场景**：用户依次打开了 Figma（设计工具）、终端（Terminal）和 VS Code。
- **AI行为**：识别出“前端开发启动”的序列特征，主动帮用户拉起本地开发服务器，并排列好窗口布局。
- **价值**：将重复的机械操作自动化，提供“懂你”的魔法体验。

### 3. 沉浸度保护与防打扰 (Focus Management)
- **场景**：用户在 IDE 或沉浸式写作软件中连续停留超过 30 分钟。
- **AI行为**：自动将系统通知静音，拦截非紧急的即时通讯消息，并在后台生成消息摘要。当用户切回通讯软件时，再进行汇总汇报。
- **价值**：保护用户的心流状态（Flow State），降低认知切换成本。

### 4. 构建个人工作记忆与数字时间轴 (Digital Memory)
- **场景**：用户询问“昨天下午我和张三开会时，查阅的那个技术文档在哪？”
- **AI行为**：结合日历数据与应用切换的时间轴记录，AI 精准回溯当时用户正在使用的浏览器页面或本地文档。
- **价值**：利用时间+行为的多模态线索，增强大模型的信息检索召回率。

## 二、 产品与交互设计：如何避免打扰？

在交互设计上，最大的挑战是**“如何做到主动智能而不显得过度打扰”**。我们需要建立一套人因效度评估机制。

### 1. 交互模式
- **隐式注入（首选）**：不主动弹窗，而是将捕获到的应用状态作为隐藏的 Context 注入到 LLM 的 Prompt 中。当用户主动唤醒 AI 时，AI 已经“知道”用户正在看什么。
- **微侵入式建议（按需）**：使用屏幕边缘的胶囊态（Pill UI）或状态栏微动效，仅在置信度极高时提供点击即用的建议。

### 2. 核心效度指标
- **建议采纳率 (Acceptance Rate)**：AI 主动发起的建议被用户点击或执行的比例（及格线 30%，优秀线 60% 以上）。
- **任务平均完成时间缩短率 (Time-to-Completion Reduction)**：衡量跨应用任务（如查资料 -> 写代码）的时效提升。
- **打扰废弃率 (Dismissal/Annoyance Rate)**：用户主动关闭 AI 提示或将其折叠的频次，该指标必须严格控制在 5% 以下。

## 三、 技术架构与算法实现

为了将原始的切换事件（Event Stream）转化为高层语义意图（Semantic Intent），存在两种实现路径：

### 路径 A：基于规则引擎与有限状态机 (Rule-based FSM)
- **原理**：定义明确的应用类别（如 IDE=开发，Chrome=浏览），通过滑动窗口记录最近 N 个动作，匹配预设的转移矩阵（如 `Browser -> IDE = 复制查阅`）。
- **优势**：极低的系统开销（CPU/内存占用几乎忽略不计）；零延迟；100% 的可解释性，不会产生幻觉。
- **劣势**：无法处理长尾场景，规则维护成本高，缺乏个性化。
- **适用阶段**：产品冷启动期，用于快速验证核心场景。

### 路径 B：基于时序行为序列的小语言模型 (Sequence Modeling via SLM)
- **原理**：将应用名称、停留时长、切换时间戳序列化（如 `[App:Safari, Duration:120s] -> [App:VSCode, Duration:10s]`），输入到本地部署的轻量级序列模型（如基于 Transformer 的 1B 级别小模型），预测用户的 Next-Action，并由大模型（LLM）进行自然语言解释。
- **优势**：能够自适应用户的个性化习惯（千人千面），发现隐性工作流，泛化能力极强。
- **劣势**：需要在本地常驻模型，存在一定的显存/内存开销；需要收集脱敏数据进行微调（Fine-tuning）。
- **适用阶段**：产品成熟期，构建技术壁垒。

> **跨域整合建议**：采用 **A + B 混合架构**。底层使用规则引擎进行事件过滤和高频确定性场景的拦截，对于复杂意图，将状态序列组装成 Context 投递给 LLM 进行动态推理。

## 四、 风险评估与 Mitigation 策略

在推动这项创新落地时，我们必须直面以下潜在矛盾：

| 潜在风险/矛盾 | 缓解策略 (Mitigation) |
|--------------|----------------------|
| **隐私安全风险 vs 数据价值** <br> 监听全局应用引发信任危机 | **Local-First 原则**：日志脱敏（仅记录进程名），绝对禁止原始行为日志上云。确需云端 LLM 推理时，只发送当前激活的单个应用上下文。 |
| **系统性能损耗 vs 实时感知** <br> 频繁轮询消耗电量 | **原生事件驱动**：摒弃轮询，采用 macOS 原生 API（如 `NSWorkspaceDidActivateApplicationNotification`），做到“无事件、零消耗”。 |
| **过度智能 vs 掌控感缺失** <br> AI 频繁自作主张惹人反感 | **白名单/黑名单机制**：允许用户配置免打扰应用。所有自动化动作遵循“首次询问并记住选择（Ask once, remember forever）”的交互原则。 |

## 五、 后续扩展行动项 (Next Steps)

1. **实现轻量级上下文注入原型 (Context Injection)**：在用户向 Copilot 提问时，底层探针隐式附带当前所处应用及上一个应用的信息。
2. **构建意图预测规则库**：梳理首批 3-5 个高频、高确定性的跨应用切换链路（如 Browser -> Terminal -> IDE）。
3. **设计状态栏微动效交互**：在不打断用户心流的前提下，完成 AI 建议的轻量级触达验证。
