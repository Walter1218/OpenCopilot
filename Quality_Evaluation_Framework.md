# OpenCopilot 划词功能质量评价框架

> **版本**: v2.0  
> **更新日期**: 2026-05-27  
> **设计理念**: 围绕划词核心交互功能设计评价体系

---

## 1. 评价理念

### 1.1 设计原则

传统评价体系（准确性、完整性、流畅性等）过于通用，无法针对划词功能的具体场景进行精准评价。

**新评价体系的核心理念**：
- **场景驱动**：每个划词功能有独立的评价维度
- **功能相关**：评价维度与功能特性直接对应
- **可操作**：每个维度都有明确的改进建议

### 1.2 划词功能与评价维度映射

| 划词功能 | 核心评价维度 | 设计理由 |
|----------|-------------|---------|
| 自动模式 | 类型判断准确性、响应恰当性、输出质量 | 自动模式需要正确判断输入类型并给出恰当响应 |
| 翻译 | 准确性、信、达、雅、术语一致性 | 翻译追求"信达雅"三境界 |
| 代码解析 | 功能总结、漏洞发现、优化建议、解释清晰度 | 代码解析需要深入理解代码逻辑和潜在问题 |
| 润色 | 语病修正、专业度提升、流畅度改善、语义保持 | 润色需要在保持原意的基础上提升质量 |
| 全文修订 | 修订质量、联动发现、矛盾检测、零误报、格式规范 | 全文修订的核心是联动分析和矛盾检测 |
| 自定义指令 | 指令遵循度、输出规范、格式保持、修改精准 | 自定义指令需要严格遵循用户指令 |

## 2. 各场景评价维度详解

### 2.1 自动模式 (auto)

**功能描述**：AI 自动判断输入类型（外语/代码/普通文本），并给出翻译/解释/总结

**评价维度**：

| 维度 | 权重 | 说明 | 评分标准 |
|------|------|------|---------|
| 类型判断准确性 | 40% | AI 是否正确判断输入类型 | 5分：完全正确；3分：基本正确；1分：判断错误 |
| 响应恰当性 | 35% | 响应是否符合类型要求 | 5分：完全恰当；3分：基本恰当；1分：不恰当 |
| 输出质量 | 25% | 输出的排版和可读性 | 5分：排版清晰；3分：基本清晰；1分：混乱 |

**评分示例**：
```
输入: "Hello World"
输出: "你好世界"
评分: 类型判断5分 + 响应恰当5分 + 输出质量5分 = 5.0/5.0
```

### 2.2 翻译 (translate)

**功能描述**：金牌翻译官，中英互译，追求信达雅

**评价维度**：

| 维度 | 权重 | 说明 | 评分标准 |
|------|------|------|---------|
| 翻译准确性 | 30% | 翻译是否准确无误 | 5分：完全准确；3分：基本准确；1分：有明显错误 |
| 信（忠实原文） | 25% | 是否忠实于原文含义 | 5分：完全忠实；3分：基本忠实；1分：偏离原意 |
| 达（表达通顺） | 20% | 表达是否通顺自然 | 5分：非常通顺；3分：基本通顺；1分：生硬 |
| 雅（用词优雅） | 15% | 用词是否优雅得体 | 5分：用词优雅；3分：用词普通；1分：用词不当 |
| 术语一致性 | 10% | 专业术语翻译是否一致 | 5分：完全一致；3分：基本一致；1分：不一致 |

**评分示例**：
```
输入: "The quick brown fox jumps over the lazy dog."
输出: "敏捷的棕色狐狸跳过了懒惰的狗。"
评分: 准确性5分 + 信5分 + 达5分 + 雅4分 + 术语5分 = 4.85/5.0
```

### 2.3 代码解析 (code)

**功能描述**：资深架构师，总结核心功能 + 指出漏洞/优化空间

**评价维度**：

| 维度 | 权重 | 说明 | 评分标准 |
|------|------|------|---------|
| 功能总结准确性 | 30% | 是否准确总结代码功能 | 5分：完全准确；3分：基本准确；1分：总结错误 |
| 漏洞发现率 | 25% | 是否发现潜在漏洞 | 5分：发现所有漏洞；3分：发现主要漏洞；1分：未发现 |
| 优化建议合理性 | 25% | 优化建议是否合理可行 | 5分：建议合理；3分：建议基本可行；1分：建议不可行 |
| 解释清晰度 | 20% | 解释是否清晰易懂 | 5分：非常清晰；3分：基本清晰；1分：难以理解 |

**评分示例**：
```
输入: "def add(a, b): return a + b"
输出: "功能：实现两数相加\n漏洞：无类型检查\n优化：添加类型提示"
评分: 功能总结5分 + 漏洞发现4分 + 优化建议5分 + 解释清晰5分 = 4.75/5.0
```

### 2.4 润色 (polish)

**功能描述**：资深编辑，修正语病，提升专业度和流畅度

**评价维度**：

| 维度 | 权重 | 说明 | 评分标准 |
|------|------|------|---------|
| 语病修正率 | 30% | 是否修正了语病 | 5分：全部修正；3分：修正大部分；1分：未修正 |
| 专业度提升 | 25% | 专业度是否提升 | 5分：明显提升；3分：有所提升；1分：未提升 |
| 流畅度改善 | 25% | 流畅度是否改善 | 5分：明显改善；3分：有所改善；1分：未改善 |
| 语义保持度 | 20% | 是否保持原意 | 5分：完全保持；3分：基本保持；1分：偏离原意 |

**评分示例**：
```
输入: "我觉得这个方案挺好的，应该没问题。"
输出: "该方案可行性较高，建议予以采纳。"
评分: 语病修正5分 + 专业度提升5分 + 流畅度改善4分 + 语义保持5分 = 4.8/5.0
```

### 2.5 全文修订 (revision)

**功能描述**：文档修订专家，修改选中文本 + 全文联动扫描

**评价维度**：

| 维度 | 权重 | 说明 | 评分标准 |
|------|------|------|---------|
| 修订质量 | 30% | 修订后的文本质量 | 5分：质量优秀；3分：质量良好；1分：质量差 |
| 联动发现率 | 25% | 是否发现联动影响 | 5分：全部发现；3分：发现主要联动；1分：未发现 |
| 矛盾检测准确性 | 20% | 是否准确检测矛盾 | 5分：完全准确；3分：基本准确；1分：有误报 |
| 零误报率 | 15% | 是否有误报 | 5分：零误报；3分：少量误报；1分：大量误报 |
| 输出格式规范性 | 10% | 是否遵循三段式格式 | 5分：完全规范；3分：基本规范；1分：格式混乱 |

**评分示例**：
```
输入选中: "项目总负责人为张伟"
输出: "📝 修订后文本：张伟担任项目总负责人，负责项目整体推进与资源协调。
🔍 联动影响分析：✅ 全文未发现需要联动修改的位置
💡 修订说明：将陈述句式调整为更自然的主动表述"
评分: 修订质量5分 + 联动发现5分 + 矛盾检测5分 + 零误报5分 + 格式规范5分 = 5.0/5.0
```

### 2.6 自定义指令 (custom)

**功能描述**：用户输入修改指令，AI 严格按指令输出结果

**评价维度**：

| 维度 | 权重 | 说明 | 评分标准 |
|------|------|------|---------|
| 指令遵循度 | 35% | 是否严格遵循指令 | 5分：完全遵循；3分：基本遵循；1分：未遵循 |
| 输出规范性 | 25% | 输出是否规范（无多余解释） | 5分：完全规范；3分：基本规范；1分：不规范 |
| 格式保持度 | 20% | 是否保持原有格式 | 5分：完全保持；3分：基本保持；1分：格式改变 |
| 修改精准度 | 20% | 是否只修改指令要求的部分 | 5分：完全精准；3分：基本精准；1分：修改过多/过少 |

**评分示例**：
```
指令: "翻译为日语"
输入: "Hello World"
输出: "ハローワールド"
评分: 指令遵循5分 + 输出规范5分 + 格式保持5分 + 修改精准5分 = 5.0/5.0
```

### 2.7 跨场景通用维度 (common)

**功能描述**：所有场景都需要评价的通用质量维度

**评价维度**：

| 维度 | 权重 | 说明 | 评分标准 |
|------|------|------|---------|
| 输出长度控制 | 40% | 输出长度是否合理 | 5分：长度合理；3分：基本合理；1分：过长或过短 |
| 错误处理能力 | 35% | 错误情况下的处理是否得当 | 5分：处理得当；3分：基本得当；1分：处理不当 |
| 多次调用一致性 | 25% | 格式和风格是否一致 | 5分：完全一致；3分：基本一致；1分：不一致 |

**评分示例**：
```
输入: "测试"
输出: "测试结果"
评分: 长度控制5分 + 错误处理5分 + 一致性5分 = 5.0/5.0
```

### 2.8 边界情况处理维度 (edge_case)

**功能描述**：处理边界情况和异常输入的能力

**评价维度**：

| 维度 | 权重 | 说明 | 评分标准 |
|------|------|------|---------|
| 空输入处理 | 35% | 空输入时是否给出适当提示 | 5分：给出提示；3分：基本处理；1分：无提示 |
| 无效输入处理 | 35% | 无效输入时是否给出错误提示 | 5分：给出提示；3分：基本处理；1分：无提示 |
| 模糊指令处理 | 30% | 模糊指令时是否要求澄清 | 5分：要求澄清；3分：合理默认；1分：盲目执行 |

**评分示例**：
```

输出: "请输入需要处理的文本"

```

## 3. 评价工具使用

### 3.1 Python API

```python
from tools.evaluation_tools import (
    evaluate_auto, evaluate_translate, evaluate_code,
    evaluate_polish, evaluate_revision, evaluate_custom
)

# 1. 评价自动模式
report = evaluate_auto(
    input_text="Hello World",
    output_text="你好世界"
)
print(f"总分: {report.total_score:.1f}/5.0 ({report.level})")

# 2. 评价翻译
report = evaluate_translate(
    input_text="The quick brown fox",
    output_text="敏捷的棕色狐狸",
    reference="那只敏捷的棕色狐狸"  # 可选
)

# 3. 评价代码解析
report = evaluate_code(
    input_code="def add(a, b): return a + b",
    output_text="功能：两数相加..."
)

# 4. 评价润色
report = evaluate_polish(
    input_text="我觉得这个方案挺好的",
    output_text="该方案可行性较高"
)

# 5. 评价全文修订
report = evaluate_revision(
    selection="项目总负责人为张伟",
    full_document="完整的项目文档...",
    output_text="📝 修订后文本：...\n🔍 联动影响分析：...\n💡 修订说明：..."
)

# 6. 评价自定义指令
report = evaluate_custom(
    instruction="翻译为日语",
    input_text="Hello World",
    output_text="ハローワールド"
)

# 7. 查看通用维度得分（自动包含在所有场景中）
print("\n通用维度得分：")
for result in report.results:
    if result.dimension in ["output_length_control", "error_handling", "consistency"]:
        print(f"  {result.dimension_label}: {result.score:.1f}/5.0")

# 8. 查看边界情况维度得分（自动包含在所有场景中）
print("\n边界情况维度得分：")
for result in report.results:
    if result.dimension in ["empty_input", "invalid_input", "ambiguous"]:
        print(f"  {result.dimension_label}: {result.score:.1f}/5.0")
```

### 3.2 查看详细结果

```python
# 查看各维度得分
for result in report.results:
    print(f"{result.dimension_label}: {result.score:.1f}/5.0")
    print(f"  反馈: {result.feedback}")
    if result.suggestions:
        print(f"  建议: {', '.join(result.suggestions)}")

# 查看总结
print(report.summary)

# 查看改进计划
print(report.improvement_plan)
```

## 5. 评价体系与 Prompt 优化

### 5.1 评价驱动的 Prompt 迭代

```
生成内容 → 质量评价 → 识别弱项 → 优化 Prompt → 再次生成 → 验证效果
```

### 5.2 各场景 Prompt 优化方向

| 场景 | 常见弱项 | Prompt 优化方向 |
|------|---------|---------------|
| 自动模式 | 类型判断错误 | 添加更明确的类型判断规则 |
| 翻译 | 雅（用词优雅）不足 | 添加"使用更书面化/优美的表达" |
| 代码解析 | 漏洞发现率低 | 添加"重点关注安全和性能问题" |
| 润色 | 语病修正不彻底 | 添加"逐句检查语法和用词" |
| 全文修订 | 联动发现率低 | 添加"扫描全文，找出所有关联位置" |
| 自定义指令 | 指令遵循度低 | 添加"严格按指令执行，不要添加额外内容" |

## 7. 总结

### 7.1 新评价体系的优势

1. **场景化**：每个划词功能有独立的评价维度
2. **功能相关**：评价维度与功能特性直接对应
3. **可操作**：每个维度都有明确的改进建议
4. **可扩展**：支持添加新的划词功能和评价维度
5. **全面性**：包含跨场景通用维度和边界情况处理维度
6. **鲁棒性**：能够评价错误处理和边界情况

### 7.2 与旧评价体系的对比

| 对比项 | 旧评价体系 | 新评价体系 |
|--------|-----------|-----------|
| 评价维度 | 通用（准确性、完整性、流畅性等） | 场景化（类型判断、信达雅、联动发现等）+ 通用维度 + 边界情况维度 |
| 适用范围 | 所有文本生成场景 | OpenCopilot 划词功能 |
| 改进建议 | 通用建议 | 针对性建议 |
| 评价精度 | 一般 | 精准 |
| 鲁棒性评价 | 无 | 包含错误处理和边界情况评价 |
| 全面性 | 基础 | 全面覆盖各种场景和情况 |
