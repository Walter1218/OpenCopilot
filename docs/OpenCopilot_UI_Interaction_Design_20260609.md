## 1. 我对 OpenCopilot 的 UI 理解

我对 `OpenCopilot` 的核心理解不是“一个 AI 聊天应用”，而是一个围绕 **右键触发、上下文就地处理、深度工作可渐进展开** 的系统级 AI 交互体系。

它的 UI 不是单窗口单形态，而是三层递进：

1. `Smart Copilot`
   - 是最短路径入口。
   - 用户在任意应用里选中内容后，双击右键唤起。
   - 核心价值是“就地理解、就地修改、就地追问”。

2. `Workspace`
   - 是复杂任务的任务中枢。
   - 用户通过三击右键进入，用来管理任务、聊天、文件、记忆和设置。
   - 核心价值是“把一次操作提升为一个持续任务”。

3. `PPT 共创模式（Studio）`
   - 是内容创作型任务的专业工作台。
   - 它不是普通 Tab 内的一个表单，而是独立共创窗口。
   - 核心价值是“把 AI 从生成器变成可协作的编辑搭子”。

因此，OpenCopilot 最合理的 UI 心智模型不是：

```text
聊天窗口 -> 所有能力都塞进一个对话框
```

而是：

```text
右键瞬时入口 -> 任务工作台 -> 专业创作工作台
Smart Copilot -> Workspace -> Studio
```

这个层级关系和当前项目代码状态是匹配的：

- `SmartCopilotV5` 是 3-Tab 浮层主壳
- `WorkspaceV5` 是 Sidebar + 5 Panel 工作台
- `StudioWindowV5` 是独立 PPT 共创窗口
- `NavigationManager` 负责三者之间的跳转与生命周期

## 2. 整体交互架构

## 2.1 用户视角

```text
任意软件中工作
   ↓
选中文字 / 选代码 / 选段落 / 想继续创作
   ↓
双击右键 -> Smart Copilot
   ↓
如果是快速理解/修订 -> 留在 Work
如果是多轮追问 -> 切到 Chat
如果是生成/共创 PPT -> 打开 Studio
如果任务变复杂 -> 三击右键进入 Workspace
```

## 2.2 产品视角

- `Smart Copilot` 负责高频、低成本、短路径操作
- `Workspace` 负责任务组织、上下文沉淀和多资料协作
- `Studio` 负责结构化内容创作和可视化编辑

## 2.3 技术视角

- 固定一套 `V5 UI`
- 统一通过 `V5AgentWorker` 触发 AI 能力
- 统一由 `NavigationManager` 编排窗口跳转
- 第三方智能体差异不直接暴露在 UI 层

这意味着 UI 的重点不是“展示哪个模型”，而是：

- AI 正在看什么
- 它准备做什么
- 它做完后用户如何确认、继续、应用

## 3. 统一交互原则

这三大窗口虽然层级不同，但应该遵循同一套交互原则。

### 3.1 原则一：就地触发

- 用户不应先切到一个 AI 软件，再把内容喂进去
- 入口要贴着当前工作对象出现
- `Smart Copilot` 是默认第一接触点

### 3.2 原则二：AI 视野透明

- 用户必须知道 AI 当前拿到的上下文是什么
- 不能让 AI 像黑盒一样“突然给答案”
- 所以 `Context Header`、`Context Strip`、`Task`、`Source Panel` 都是必要设计

### 3.3 原则三：短任务和长任务分层

- 快速解释、修复、润色属于短任务
- 研究、持续聊天、资料编排属于长任务
- PPT 创作属于专业任务

因此不应把三类任务都塞进同一个聊天面板。

### 3.4 原则四：AI 改动可确认

- 对高影响操作，优先给预览或差异确认
- 用户始终拥有最终控制权
- 这与 OpenCopilot 的“审查修正”定位一致

### 3.5 原则五：跨窗口要自然接力

- `Work` 结果应能自然进入 `Chat`
- `Workspace` 中定义的任务应影响后续 AI 行为
- `Studio` 的输出应能回到 `Chat` 或 `Workspace`

│                  OpenCopilot                       │

│  Layer 1  Smart Copilot                           │

> `Smart Copilot` 负责“此刻这一段”，`Workspace` 负责“这一整个任务”，`Studio` 负责“这份最终作品”。

## 5. 核心窗口一：Smart Copilot

## 5.1 角色定位

`Smart Copilot` 是 OpenCopilot 最核心的系统级入口。

它的价值不是“功能最多”，而是：

- 出现最快
- 离当前内容最近
- 决策成本最低

当前项目里，它最适合承担三类动作：

1. 快速理解
2. 快速修正
3. 快速转入更深操作

## 5.2 当前结构理解

当前代码形态已经很清晰：

- 浮层窗口
- `680 x 520`
- 3 个 Tab
  - `Work`
  - `Chat`
  - `Studio`

这不是传统的“一个悬浮聊天框”，而是一个 **右键短任务控制台**。

## 5.3 交互设计稿

```text
┌──────────────────────────────────────────────────────┐
│ ✨ Smart Copilot                         ●  ⚙  ✕     │
├──────────────────────────────────────────────────────┤
│ Tabs: [⚡ Work] [💬 Chat] [🎨 Studio]                 │
├──────────────────────────────────────────────────────┤
│ Header                                              │
│ 📎 [Selection] 236 字符 · Word / VS Code / Browser   │
├──────────────────────────────────────────────────────┤
│ Context Strip                                       │
│ [Selection] [Doc] [Browser] [Clipboard] [File]      │
├──────────────────────────────────────────────────────┤
│ Primary Actions                                     │
│ [Explain]      [Fix]      [Polish]                  │
├──────────────────────────────────────────────────────┤
│ Secondary Actions                                   │
│ [Translate] [Code Review] [More] [语言选择]          │
├──────────────────────────────────────────────────────┤
│ Result Area                                         │
│ - 当前结果 / 流式输出 / 结构化建议                   │
│ - 可展示“已获取内容预览”或“正在处理”                │
├──────────────────────────────────────────────────────┤
│ Action Bar                                          │
│                                [Copy] [Export] [Apply]│
└──────────────────────────────────────────────────────┘
```

## 5.4 我建议的交互重点

### A. `Work` 是默认首页

原因：

- 右键唤起后的用户心智是“帮我处理这段”
- 不是“我要开一段长期对话”
- 所以默认应该直达操作，而不是先问问题

### B. Header 必须表达“AI 正在看什么”

当前项目已经有这个方向，但我建议对外设计稿明确为：

- 来源类型：`Selection / Doc / Browser / Clipboard / File`
- 来源宿主：如 `Word / PowerPoint / Chrome / VS Code`
- 内容规模：如 `236 字符`
- 在线状态：如 `Online / Degraded`

### C. Primary / Secondary 分层是对的

这是一个很关键的设计判断。

推荐继续保持：

- 高频动作大按钮：`Explain / Fix / Polish`
- 次高频动作小按钮：`Translate / Code Review / More`

因为用户的第一步决策越少，浮层越“像系统能力”，而不是“像另一个复杂 App”。

### D. `Chat` 不应替代 `Work`

`Chat` 在 `Smart Copilot` 中的职责不是主入口，而是：

- 承接追问
- 承接复杂解释
- 承接上下文延展

所以这里的 `Chat` 更像“在当前任务上继续说两句”，不是独立会话中心。

### E. `Studio` 在这里是入口，不是主编辑区

`Smart Copilot` 里的 `Studio` Tab 更适合理解为：

- 一个“启动共创”的入口页
- 一个“快速创建”的轻交互区
- 一个“共创状态提示”区域

而不应该把完整 PPT 编辑器塞回这个浮层里。

## 5.5 `Smart Copilot` 的关键交互流

### 流程 1：选中内容 -> 快速解释

```text
用户在任意应用中选中内容
  -> 双击右键
  -> Smart Copilot 出现
  -> Work 默认展示选区摘要
  -> 用户点击 Explain
  -> Result Area 流式输出
  -> 用户 Copy / Apply / 继续去 Chat
```

### 流程 2：快速处理 -> 转入对话

```text
用户在 Work 中完成第一次 Explain / Fix
  -> 发现需要继续追问
  -> 点击 Continue in Chat 或切换 Chat
  -> 系统自动注入上下文摘要
  -> 用户继续对话
```

### 流程 3：从当前内容发起 PPT 共创

```text
用户在 Word / 浏览器 / 研究资料中选中内容
  -> Smart Copilot 打开
  -> 切到 Studio
  -> 快速创建 / 打开 Studio
  -> 独立 StudioWindowV5 打开
```

## 5.6 用户层评价指标

- 唤起后用户是否能在 1 秒内知道 AI 正在看什么
- 首次操作是否能在 1 次点击内完成
- 从 `Work` 转 `Chat` 是否无需重新解释上下文
- 从 `Smart Copilot` 转 `Studio` 是否保持任务连续感

## 6. 核心窗口二：Workspace

## 6.1 角色定位

`Workspace` 不是大号聊天窗口，而是一个 **任务编排与上下文组织中心**。

我对它的理解是：

- `Smart Copilot` 解决“局部问题”
- `Workspace` 解决“整件事”

如果说 `Smart Copilot` 是“右键参谋”，那么 `Workspace` 是“任务驾驶舱”。

## 6.2 当前结构理解

当前代码里，`WorkspaceV5` 已经非常接近成熟工作台形态：

- 独立窗口
- `1000 x 700`
- 左侧 Sidebar
- 右侧 5 个 Panel
  - `Task`
  - `Chat`
  - `Files`
  - `Memory`
  - `Settings`

这个结构是合理的，因为复杂任务至少需要回答 5 个问题：

1. 我在做什么任务
2. 我和 AI 聊到了哪里
3. 我有哪些资料
4. 系统知道哪些长期记忆
5. 当前配置是什么

## 6.3 交互设计稿

```text
┌──────────────────────────────────────────────────────────────┐
│ 🤖 Agent Workspace                                    ✕     │
├───────────────┬──────────────────────────────────────────────┤
│ Sidebar       │ Main Panel                                  │
│               │                                              │
│ [Task]        │  Panel Header                                │
│ [Chat]        │  当前面板标题 + 状态 + 操作                  │
│ [Files]       │                                              │
│ [Memory]      │  Panel Content                               │
│ [Settings]    │  - Task: 任务定义 / 模板 / 历史              │
│               │  - Chat: 会话列表 / 对话 / 输入              │
│               │  - Files: 文件筛选 / 预览 / 发送              │
│               │  - Memory: 知识图谱 / 术语 / 记忆注入         │
│               │  - Settings: 配置摘要 / 跳转设置             │
│               │                                              │
│  ⚙ Settings   │                                              │
│  ✕ Close      │                                              │
└───────────────┴──────────────────────────────────────────────┘
```

## 6.4 五个 Panel 的最佳交互定义

### A. `Task`

这是 `Workspace` 的第一入口，不是装饰页。

职责：

- 定义当前任务
- 让用户明确“这次 AI 帮我的边界是什么”
- 给后续聊天和资料操作一个统一背景

理想感知：

> 我不是在和 AI 闲聊，我是在推进一个明确任务。

建议保留的核心交互：

- 任务描述输入
- 模板加载
- 从剪贴板/最近文件导入
- 历史任务恢复

### B. `Chat`

这里的 `Chat` 和 `Smart Copilot` 里的 `Chat` 不同。

这里它是：

- 任务级对话中心
- 多轮协作主战场
- 多意图能力入口

建议理解为“任务会话”而不是“普通聊天”。

### C. `Files`

这是 `Workspace` 能体现生产力差异的关键面板。

职责：

- 让资料进入任务上下文
- 让资料在窗口内完成预览、发送、跳转

关键动作建议保留：

- 文件筛选
- 文件预览
- 发送到 `Task`
- 发送到 `Chat`
- 发送到 `Studio`
- 复制内容 / 复制路径

### D. `Memory`

这是 OpenCopilot 区别于普通 AI 聊天应用的关键面板。

这里不应该只是“展示数据库里有什么”，而应该让用户感知：

- 系统有什么长期知识
- 哪些记忆可被本次任务引用
- 这些知识如何提升回答质量

### E. `Settings`

在 `Workspace` 中，`Settings` 更适合做“配置摘要面板”，而不是完整配置编辑器。

也就是：

- 在这里看全局状态
- 在这里做轻运维动作
- 真正修改时打开统一设置弹窗

## 6.5 `Workspace` 的关键交互流

### 流程 1：从模糊目标到明确任务

```text
用户三击右键进入 Workspace
  -> 打开 Task
  -> 输入任务背景
  -> 任务被保存为当前上下文
  -> 后续 Chat / Files / Memory 都围绕该任务展开
```

### 流程 2：文件进入任务

```text
用户进入 Files
  -> 选择最近文件或拖入新文件
  -> 预览文件内容
  -> 点击“发送到 Chat / Task / Studio”
  -> 文件内容成为后续 AI 上下文的一部分
```

### 流程 3：工作台内研究 -> 输出到创作

```text
用户在 Workspace Chat 中完成研究和信息整理
  -> 发现需要做 PPT
  -> 将整理结果发送到 Studio
  -> Studio 打开并承接当前任务内容
```

## 6.6 与 `Smart Copilot` 的关系

`Workspace` 不是 `Smart Copilot` 的替代，而是其“深水区”。

最自然的关系应该是：

- `Smart Copilot` 负责瞬时操作
- `Workspace` 负责把这些瞬时操作沉淀成任务

推荐用一句设计语言概括：

> `Smart Copilot` 处理片段，`Workspace` 组织全局。

## 6.7 用户层评价指标

- 用户是否能在 10 秒内建立一个清晰任务
- 文件是否能无摩擦进入任务上下文
- `Memory` 是否让用户感知到“系统知道我什么”
- 用户是否会把它当“工作台”而不是“第二个聊天框”

## 7. 核心窗口三：PPT 共创模式（Studio）

## 7.1 角色定位

`Studio` 是 OpenCopilot 在“生成之后的深加工”阶段最重要的专业窗口。

它不是一个“帮你一键生成 PPT 的按钮”，而是一个：

- 可看原文
- 可调大纲
- 可看预览
- 可和 AI 协商修改

的共创工作台。

这与项目整体定位非常一致：

> OpenCopilot 的价值不是替你一把做完，而是让你把关、让 AI 配合。

## 7.2 当前结构理解

当前项目中，`StudioWindowV5` 已具备明确的 4 区模型：

- `Source`
- `Outline`
- `Preview`
- `底部 AI Chat`

这已经是一个非常合理的专业创作界面。

它说明产品对 PPT 的理解不是“结果下载页”，而是“共创编辑器”。

## 7.3 交互设计稿

```text
┌────────────────────────────────────────────────────────────────────┐
│ 🚦 PPT 人机共创工作台                 幻灯片:8 要点:24 原文:67%  ✕ │
├────────────────────────────────────────────────────────────────────┤
│ Source           │ Outline / Thumbs        │ Preview              │
│                  │                         │                      │
│ 原文全文         │ 缩略图 / 页列表         │ 当前页可视化预览     │
│ 高亮已使用段落   │ 标题 / 版式 / 要点编辑   │ 差异预览 / 点击编辑   │
│ 点击段落跳转页   │ 页间切换                 │ 当前页查看与确认     │
├────────────────────────────────────────────────────────────────────┤
│ AI 共创区                                                         │
│ 🤖 输入指令：把第 2 页标题更短一些 / 把这段数据改成图表            │
│                                        [发送] [撤销] [重做]       │
├────────────────────────────────────────────────────────────────────┤
│ 主题色块                                     [全屏预览] [导出 PPT] │
└────────────────────────────────────────────────────────────────────┘
```

## 7.4 这个界面为什么成立

`Studio` 的关键不是“好看”，而是把 4 个创作问题拆开：

### A. `Source`

回答：

> 原始内容是什么？AI 用了哪些？漏了哪些？

这是信任建立的来源。

### B. `Outline`

回答：

> 结构是否合理？页次是否清晰？要点是否完整？

这是结构编辑区。

### C. `Preview`

回答：

> 当前这页看起来到底是什么样？

这是成果感知区。

### D. `AI Chat`

回答：

> 我想修改，但不一定想手动改所有字段。

这是协商入口，不是主界面本身。

## 7.5 我理解的最佳交互心流

### 阶段 1：带着内容进入

来源可以是：

- `Smart Copilot` 当前选中内容
- `Workspace` 整理好的研究结果
- 粘贴输入
- 文件导入

用户的感觉应该是：

> 我不是从零开始，而是带着现成上下文进入共创。

### 阶段 2：先看结构，再看页面

进入 Studio 后，不应该先让用户面对一堆参数。

最自然的顺序是：

1. 先看到整体页数和大纲
2. 再切具体页预览
3. 再决定手改还是让 AI 改

### 阶段 3：AI 修改以差异确认收口

这里是整套交互里最重要的信任设计之一。

建议始终强化：

- AI 提建议
- 用户看差异
- 用户确认是否应用

而不是“用户一句话，界面全改了”。

## 7.6 建议强调的三种典型模式

### 模式 1：原文驱动

```text
长文输入
  -> AI 生成初版
  -> 用户看 Source 和 Outline
  -> 调整结构
  -> 导出
```

适合：

- 研究报告转 PPT
- 方案文档转汇报页

### 模式 2：预览驱动

```text
用户看到某一页不满意
  -> 直接在 Preview 中查看问题
  -> 去 Outline 或 AI Chat 改
  -> 预览即时更新
```

适合：

- 调整标题长度
- 改页内布局
- 优化单页表达

### 模式 3：AI 协商驱动

```text
用户知道方向，但不想手动改字段
  -> 在 AI 区输入自然语言指令
  -> 系统生成差异建议
  -> 用户接受 / 拒绝
```

适合：

- 风格统一
- 图表替换
- 页间内容搬移

## 7.7 用户层评价指标

- 用户是否能理解“原文 -> 大纲 -> 预览 -> 导出”的关系
- AI 修改是否足够可控
- 用户是否愿意在 Studio 中停留并打磨，而非导出后到 PowerPoint 再返工
- 单页修改是否足够快，不打断创作流

---

## 8. 三大窗口的接力关系

这是我认为最关键的一部分。OpenCopilot 的 UI 价值不在单个窗口，而在三者之间的接力。

## 8.1 接力图

```text
任意应用
  -> Smart Copilot
     -> Work 直接处理
     -> Chat 继续追问
     -> Studio 进入 PPT 共创
  -> Workspace
     -> Task 定义全局目标
     -> Files / Memory 补上下文
     -> Chat 做持续协作
     -> Studio 输出最终 PPT
```

## 8.2 最重要的四条链路

### 链路 A：`Work -> Chat`

适用于：

- 第一次结果不够
- 需要多轮澄清
- 需要复杂约束

核心要求：

- 自动带入刚才上下文
- 不让用户重复描述

### 链路 B：`Smart Copilot -> Workspace`

适用于：

- 单点操作演变为复杂任务
- 用户开始需要文件、记忆、任务管理

核心要求：

- 当前选区、结果、意图能被沉淀为任务

### 链路 C：`Smart Copilot / Workspace -> Studio`

适用于：

- 需要形成正式产物
- 需要做 PPT 结构化表达

核心要求：

- 输入上下文自动传递
- 不让用户再次粘贴

### 链路 D：`Studio -> Chat`

适用于：

- 导出后继续讨论
- 导出前发现要补背景或修改措辞

核心要求：

- 带着当前文件结果回到对话

## 10. 两种可选产品叙事路径对比

为了后续对外表达更统一，这里给出两个 UI 叙事版本。

## 10.1 方案 A：强调右键效率

话术重点：

- 双击右键马上得到 AI
- 就地处理当前内容
- 不离开宿主应用

优点：

- 简洁
- 易懂
- 适合 Demo 和传播

风险：

- 容易低估 `Workspace` 和 `Studio` 的价值

## 10.2 方案 B：强调三层工作流

话术重点：

- `Smart Copilot` 负责即时协助
- `Workspace` 负责复杂任务
- `Studio` 负责最终创作

优点：

- 更完整表达产品结构
- 更适合面向投资人、产品团队、设计团队讨论

风险：

- 传播成本略高
- 初次理解门槛稍高

## 10.3 我的建议

对外展示优先用 `方案 A`，对内设计和路线图统一用 `方案 B`。

也就是：

- 对外先讲“右键即用”
- 对内完整理解“三窗口接力”

## 12. 最终结论

基于当前项目的设计和实现，我认为 `OpenCopilot` 最合理、最有辨识度的 UI 交互模型可以概括为：

> 一个由 `Smart Copilot`、`Workspace`、`Studio` 三层构成的系统级 AI 交互体系。

其中：

- `Smart Copilot` 是右键触发的瞬时入口，解决当前片段
- `Workspace` 是复杂任务的组织中枢，管理上下文、资料与会话
- `Studio` 是专业产出的共创工作台，负责把内容打磨成可交付作品

如果只用一句话概括这套交互设计，我会写成：

> `Smart Copilot` 解决“这里”，`Workspace` 解决“这件事”，`Studio` 解决“这个结果”。
