# PPT 忠实改写专项评测规范

更新日期：`2026-06-11`

## 目标

- 建立一套专门评估 `PPT 共创忠实改写能力` 的标准，而不是把这类能力混在泛化的“文案润色”里。
- 评估重点不是“写得更漂亮”，而是：
  - 在更专业的表达下，是否仍然忠实保留原始事实
  - 是否保持原有结构、顺序和语义关系
  - 是否避免幻觉、偷换结论和风险弱化
- 这套专项评测直接服务于 `prompt 迭代闭环`，用于比较不同 prompt 版本的收益与回归。

## 适用场景

- `ppt_editor` 当前页文案润色
- 董事会/高层汇报风格改写
- 压缩改写、结论化改写、专业化改写
- 任何包含“更专业/更正式/更像汇报材料，但不要改事实”的共创需求

## 能力定义

### 1. 事实保留率 `Fact Retention`

- 数字必须保留：
  - 金额
  - 百分比
  - 年份 / 时间点
  - 人数 / 数量 / 区间
- 专有名词必须保留：
  - 产品名
  - 公司名
  - 组织名
  - 英文缩写
  - 协议名
- 明确状态必须保留：
  - 已完成
  - 计划中
  - 风险
  - 限制
  - 假设

### 2. 结构保持度 `Structure Preservation`

- 原有条目数量尽量不变
- 原有条目顺序尽量不变
- 不随意合并多条结论
- 不把一条拆成多条
- 不把标题改写误落到正文区

### 3. 专业表达度 `Professional Expression`

- 允许升级措辞和句式
- 允许压缩冗余表述
- 允许改成结论句、汇报句、董事会风格
- 但不能以“更专业”为名牺牲事实精度

### 4. 幻觉控制 `Hallucination Control`

- 不新增原文没有的事实
- 不把计划写成结果
- 不把试点写成规模化落地
- 不把风险弱化成空泛积极表达
- 不把区间数据压成单点值

## 评分标准

建议把忠实改写分成 4 个一级维度，每个维度 `0-5` 分。

### A. 事实保留率 `35%`

- `5`: 关键数字、专名、状态、关系全部保留
- `4`: 仅有轻微措辞偏移，不影响事实
- `3`: 有少量事实锚点缺失，但核心结论仍对
- `2`: 多个关键事实缺失或被模糊
- `1`: 事实明显漂移
- `0`: 大量幻觉或核心事实反转

### B. 结构保持度 `20%`

- `5`: 条目数量、顺序、粒度均保持
- `4`: 有轻微重组，但仍可一一映射
- `3`: 出现条目合并/拆分，但不影响主要内容
- `2`: 结构重排明显，映射困难
- `1`: 结构严重偏移
- `0`: 基本变成自由生成

### C. 专业表达度 `20%`

- `5`: 更正式、更清晰、更适合汇报
- `4`: 明显更专业，但仍有少量口语残留
- `3`: 略有改进
- `2`: 改写有限或表达变僵硬
- `1`: 基本无提升
- `0`: 语言质量明显退化

### D. 幻觉控制 `25%`

- `5`: 无新增事实、无偷换、无风险弱化
- `4`: 有轻微扩写，但不改变事实
- `3`: 有个别模糊化倾向
- `2`: 有明显新增推断或状态漂移
- `1`: 多处幻觉或事实包装
- `0`: 输出不可用

## 自动评测代理指标

当前代码链路中，建议把下列指标作为专项代理分：

- `description_accuracy`
  - 用于衡量“是否忠实执行约束”
- `semantic_similarity`
  - 用于衡量“改写后是否仍贴近原任务语义”
- `judge_score`
  - 用于判断是否存在事实漂移、空泛表达、结构破坏
- `current_page_hit`
  - 确认改写仍命中当前页
- `type_match`
  - 确认返回的是可执行编辑结果而不是闲聊或占位回复

建议后续继续新增 4 个专项自动指标：

- `fact_anchor_coverage`
- `structure_keep_rate`
- `risk_retention_rate`
- `hallucination_rate`

## 测试集设计原则

### Case 维度覆盖

测试集必须覆盖以下高风险改写点：

1. `数字保留`
2. `专有名词保留`
3. `状态保留`
4. `计划/结果区分`
5. `风险与限制保留`
6. `因果关系保留`
7. `顺序与粒度保持`
8. `区间/范围保留`
9. `结论句式但保事实锚点`
10. `多约束联合改写`

### Case 分层

- `L1 基础忠实改写`
  - 单一约束，如数字保留
- `L2 多约束忠实改写`
  - 数字 + 顺序 + 专名同时保留
- `L3 高风险忠实改写`
  - 计划/结果区分、风险不可弱化、负面信息保留

## 使用规范

### 跑专项 benchmark

```bash
OPEN_COPILOT_PPT_TASK_MODE=faithful_rewrite \
OPEN_COPILOT_ENABLE_LLM_JUDGE=1 \
OPEN_COPILOT_LLM_JUDGE_MAX_CASES=12 \
OPEN_COPILOT_PPT_BENCH_BACKENDS=self_agent \
python tests/e2e/test_ppt_cocreation_quality_benchmark.py
```

### 适合 prompt 迭代的原因

- case 固定
- 文档固定
- 当前页固定
- 指令固定
- 可以直接做 prompt `v1/v2/v3` 的前后对比

这意味着：

- 你可以开始进入 `prompt 迭代`
- 但前提是所有新 prompt 都必须在这套基准上做回归

## Prompt 迭代流程

建议固定以下闭环：

1. 定义本轮要优化的点
   - 例如：`专业化且保事实`
2. 只改一类 prompt 策略
   - 例如：新增“每条必须保留一个事实锚点”
3. 在 `faithful_rewrite` 数据集上全量回归
4. 看 4 类结果
   - 均分是否上升
   - 高风险 case 是否回归
   - 延迟是否异常上升
   - 是否伤害其他共创大类
5. 只有在“专项提升且无明显回归”时才合并到默认 prompt

## 准入门槛

建议把以下条件作为 prompt 升级准入标准：

- `faithful_rewrite` 平均分不低于上一版
- `事实保留率` 高风险 case 不出现明显回归
- `计划/结果区分`、`风险保留`、`数字保留` 三类 case 不允许出现 `P0` 回归
- `当前页命中` 不低于 `95%`
- 平均延迟增幅不超过 `15%`

## 不建议的做法

- 只凭单条 case 决定 prompt 好坏
- 只看 LLM 主观感觉，不看固定 benchmark
- 为了更专业而放松事实保留要求
- 在没有回归数据的情况下直接替换线上默认 prompt

## 结论

一旦这套：

- 评测标准
- 数据集
- 固定 benchmark 入口

都具备了，就可以进入系统化的 `prompt 迭代` 阶段。

更准确地说，不是“可以开始随便调 prompt”，而是：

- 现在已经具备了 `可验证、可回归、可比较` 的 prompt 迭代基础设施。
